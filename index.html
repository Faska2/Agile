Méthodes Agile et Scrum

Q1 : Quels sont les inconvénients des méthodes prévisibles : Cascade, Y et V ?

Les inconvénients des méthodes prévisibles (Waterfall/Cascade, Y, V) sont :

  • Manque de flexibilité : difficile de modifier les exigences une fois le développement commencé
  • Livraison tardive du produit fonctionnel : le client ne voit le résultat qu'à la fin
  • Risque élevé : si une erreur est détectée tardivement, elle est coûteuse à corriger
  • Communication limitée avec le client pendant le développement
  • Adaptation difficile aux changements technologiques ou de marché

Q2 : Quels sont les avantages des méthodes agiles ?

Les avantages des méthodes agiles sont :

  • Flexibilité et adaptabilité aux changements
  • Livraisons fréquentes et incrémentales de fonctionnalités
  • Collaboration étroite avec le client et les parties prenantes
  • Réduction des risques grâce aux retours rapides
  • Amélioration continue du processus et du produit
  • Équipes auto-organisées et autonomes
  • Satisfaction client accrue

Q3 : Définir la méthode Agile

La méthode Agile est une approche de gestion de projet et de développement de logiciel qui met l'accent sur la flexibilité, la collaboration client, la livraison rapide de valeur et l'adaptation aux changements. Elle repose sur 4 valeurs fondamentales :

  • Les individus et leurs interactions plus que les processus et les outils
  • Des logiciels fonctionnels plus qu'une documentation exhaustive
  • La collaboration avec les clients plus que la négociation contractuelle
  • L'adaptation au changement plus que le suivi d'un plan

Q4 : Définir un sprint

Un sprint est une période de temps fixe (généralement de 1 à 4 semaines) pendant laquelle l'équipe Scrum travaille pour accomplir un ensemble de travail défini et convenu. À la fin de chaque sprint, l'équipe devrait avoir livré un incrément de produit potentiellement livrable.

Q5 : Quelles sont les rôles dans Scrum ?

Les rôles dans Scrum sont :

  • Product Owner : Responsable de maximiser la valeur du produit, gère le Product Backlog
  • Scrum Master : Facilitateur qui s'assure que l'équipe suit les principes et pratiques Scrum
  • Équipe de développement : Groupe de professionnels qui réalisent le travail
  • Les parties prenantes : Personnes intéressées par le produit mais ne faisant pas directement partie de l'équipe Scrum

Q6 : Quel est le rôle de Scrum Master ?

Le Scrum Master est un servant-leader qui :

  • Facilite les événements Scrum
  • Élimine les obstacles qui gênent l'équipe
  • Coache l'équipe dans les pratiques Scrum
  • Protège l'équipe des distractions externes
  • Aide à l'amélioration continue
  • S'assure que l'équipe reste productive et focalisée

Q7 : Quelles sont les événements (étapes) dans Scrum ?

Les événements Scrum sont :

  • Sprint Planning : Planification du travail à réaliser pendant le sprint
  • Daily Scrum : Réunion quotidienne de 15 minutes pour synchroniser l'équipe
  • Sprint Review : Présentation du travail accompli aux parties prenantes
  • Sprint Retrospective : Réflexion sur le processus pour s'améliorer
  • Le Sprint lui-même : La période de temps pour réaliser le travail

Q8 : Quels sont les 3 Artéfacts (produits) Scrum ?

Les 3 artefacts Scrum sont :

  • Product Backlog : Liste priorisée de tout ce qui pourrait être nécessaire au produit
  • Sprint Backlog : Ensemble des éléments du Product Backlog sélectionnés pour un sprint
  • Incrément de produit : Somme de tous les éléments du Product Backlog terminés jusqu'à présent

Q9 : Donner des exemples de User Stories

Exemples de User Stories :

  • "En tant qu'utilisateur, je veux pouvoir me connecter avec mon email et mon mot de passe pour accéder à mon compte"
  • "En tant qu'administrateur, je veux pouvoir exporter les données des utilisateurs au format CSV pour les analyser"
  • "En tant que client, je veux pouvoir suivre ma commande en temps réel pour savoir quand elle sera livrée"
  • "En tant qu'acheteur, je veux pouvoir ajouter des articles à mon panier pour les acheter plus tard"

Jira

Q10 : Quel est le but de Jira ?

Jira est une plateforme de gestion de projet et de suivi des problèmes développée par Atlassian. Elle permet de :

  • Planifier, suivre et gérer les projets agiles
  • Organiser le travail en sprints et backlogs
  • Suivre les bugs et les tâches
  • Collaborer en équipe
  • Générer des rapports sur l'avancement des projets

Q11 : Définir : ticket, Epic, story, tâche, bug, BackLog

  • Ticket : Élément de travail dans Jira qui représente une tâche, un bug, une histoire, etc.
  • Epic (Épopée) : Gros travail qui peut être décomposé en plusieurs stories ou tâches plus petites
  • Story : Fonctionnalité du point de vue de l'utilisateur, généralement formulée comme une User Story
  • Tâche : Travail technique à réaliser pour compléter une story
  • Bug : Erreur ou problème dans le code qui doit être corrigé
  • BackLog : Liste priorisée de tout le travail à réaliser dans un projet

Q12 : Quels sont les types de ticket ?

Les types de ticket courants dans Jira sont :

  • Tâche (Task) : Action technique à réaliser
  • Bug : Erreur à corriger
  • Story : Fonctionnalité à réaliser du point de vue utilisateur
  • Epic : Grande fonctionnalité regroupant plusieurs stories
  • Sous-tâche (Sub-task) : Partie d'une tâche ou d'une story plus grande

Q13 : Quelles sont les étapes d'un ticket ? (à faire - en cours - terminé)

Les étapes typiques d'un ticket dans Jira sont :

  • À faire (To Do) : Le ticket est créé mais pas encore commencé
  • En cours (In Progress) : Quelqu'un travaille activement sur le ticket
  • En révision (In Review) : Le travail est terminé et en attente de validation
  • Terminé (Done) : Le travail est validé et complété

Q14 : Quel est le rôle du BackLog dans Jira Software ?

Le BackLog dans Jira Software est :

  • L'ensemble des sprints prévus pour un projet
  • Une liste priorisée de tout le travail à réaliser
  • Un espace où le Product Owner gère et priorise les éléments
  • La source de travail pour les sprints futurs
  • Un outil de planification visuel pour l'équipe

Git et GitHub

Q15 : Quel est le but de Git ?

Git est un système de contrôle de version distribué qui permet de :

  • Suivre les modifications dans le code source
  • Travailler en collaboration sur des projets
  • Garder un historique complet des changements
  • Créer des branches pour développer des fonctionnalités séparément
  • Revenir à des versions précédentes si nécessaire
  • Fusionner les modifications de plusieurs développeurs

Q16 : Quel est le but de GitHub ?

GitHub est une plateforme d'hébergement de code qui permet de :

  • Héberger des dépôts Git en ligne
  • Collaborer avec d'autres développeurs
  • Suivre et gérer les problèmes (issues)
  • Gérer les demandes de fusion (pull requests)
  • Mettre en place des intégrations continues (CI/CD)
  • Créer des pages web pour les projets
  • Participer à des projets open source

Q17 : Expliquer les commandes Git de base

# Créer un dossier
mkdir nom_du_dossier

# Se placer dans le dossier créé
cd nom_du_dossier

# Initialiser le dépôt local
git init

# Créer des fichiers
touch fichier1.txt fichier2.txt

# Afficher l'état de la sauvegarde
git status

# Ajouter les fichiers à l'index local
git add fichier1.txt fichier2.txt
# ou pour ajouter tous les fichiers
git add .

# Valider les fichiers dans le repo local avec un message
git commit -m "Message de commit descriptif"

# Créer une branche css
git branch css

# Lister les branches créées
git branch

# Basculer sur la branche css
git checkout css

# Fusionner les branches dans la branche principale master
git checkout master
git merge css

# Saisir les informations du compte GitHub
git config --global user.name "Votre Nom"
git config --global user.email "votre.email@example.com"

# Pousser l'index local vers un repo distant dans GitHub
git remote add origin https://github.com/userName/ProjetGitHub.git
git push -u origin master

# Copier le repo à partir de GitHub vers un dossier local
git clone https://github.com/userName/ProjetGitHub.git

# Afficher l'historique des commits
git log

# Afficher les modifications faites dans une branche
git diff nom_branche

# Importer un contenu de GitHub et le fusionner dans un contenu local
git pull origin master

# Supprimer une branche
git branch -d nom_branche

# Supprimer un fichier du repo local (index)
git rm nom_fichier
git commit -m "Suppression du fichier nom_fichier"

DevOps

Q18 : Quel est le rôle de DevOps ?

DevOps est une approche qui combine le développement (Dev) et les opérations (Ops) pour :

  • Accélérer le cycle de vie du développement
  • Améliorer la collaboration entre les équipes de développement et d'exploitation
  • Automatiser les processus d'intégration, de test et de déploiement
  • Assurer une livraison continue de valeur aux clients
  • Réduire les erreurs humaines grâce à l'automatisation
  • Améliorer la qualité et la fiabilité des logiciels

Q19 : Quel est le rôle de Kubernetes, Jira/Trello, Jenkins, Git, GitHub, runner ?

  • Kubernetes : Système d'orchestration de conteneurs qui automatise le déploiement, la mise à l'échelle et la gestion des applications conteneurisées
  • Jira/Trello : Outils de gestion de projet et de suivi des tâches pour organiser le travail et collaborer en équipe
  • Jenkins : Serveur d'automatisation qui permet de construire, tester et déployer des logiciels (outil d'intégration continue)
  • Git : Système de contrôle de version pour suivre les changements dans le code source
  • GitHub : Plateforme d'hébergement de code basée sur Git pour la collaboration et la gestion de projet
  • Runner : Outil qui exécute des tâches automatisées dans un pipeline CI/CD (par exemple, GitLab Runner)

Q20 : Définir : CI/CD, pipeline

  • CI/CD (Continuous Integration/Continuous Delivery or Deployment) :
    • Intégration Continue (CI) : Pratique où les développeurs intègrent régulièrement leur code dans un dépôt central, après quoi des builds et des tests automatisés sont exécutés
    • Livraison/Déploiement Continu (CD) : Pratique où les changements de code sont automatiquement préparés pour une release en production
  • Pipeline : Ensemble de processus automatisés qui permettent de déplacer le logiciel du développement à la production. Un pipeline CI/CD typique inclut des étapes de build, de test, et de déploiement automatisées.
